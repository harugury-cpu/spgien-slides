# spigen-slides Planning Guide (v6)

## 역할 분리

| 주체 | 역할 |
|------|------|
| 사용자 | 내용 제공, 구성 승인 |
| AI | 내용 정리, 슬라이드 구성 결정, 생성 |

AI는 내용을 창작하지 않는다. 사용자가 준 내용을 정리할 뿐이다.
컴포넌트는 내용을 강요하지 않는다. 형식보다 내용 보존이 우선이다.

---

## Step 1. 내용 수집

### 경로 A: 원본 데이터 있음

문서, 보드 데이터, 채팅 기록, 노트 등 원본이 있으면:
1. 원본을 읽는다
2. 핵심 사실(fact)만 추출한다
3. 추출 결과를 사용자에게 확인받는다

추출 원칙:
- 원본에 없는 내용을 추가하지 않는다
- 수치, 고유명사, 시스템명은 원본 그대로 사용한다
- 요약할 때 의미가 달라지면 원본 문장을 그대로 쓴다

### 경로 B: 사용자 직접 제공

아래 4가지를 묻는다. 이것만 묻는다.

```
Q1. 이 PPT의 목적은 무엇인가요?
    보고 / 교육·안내 / 제안 / 홍보 중 선택 또는 직접 설명

Q2. 청중은 누구인가요?
    (예: 경영진, 팀 내부, 외부 고객)

Q3. 전달할 핵심 내용을 적어주세요.
    형식 무관 — 글머리, 메모, 흐름 설명, 키워드 나열 모두 가능

Q4. 테마: 생략 가능 (기본 = light, 필요 시 dark)
```

내용(Q3)이 충분히 구체적이면 바로 Step 2로 진행한다.
테마를 따로 지정하지 않으면 **light를 기본값**으로 사용한다.

기본 deck mode는 별도 요청이 없으면 아래 **하나로 고정**한다.

```txt
operational_detail_report
```

즉:
- 운영 전달용
- 한 페이지 자기완결 설명형
- 요약형 피치덱보다 보고서형 서술 우선

동시에 아래 내부 파라미터를 기본값으로 사용한다.

```txt
detail_level = 8
layout_variance = 3
content_preservation_priority = max
execution_clarity_priority = max
```

의미:
- `detail_level=8`: 한 장에 충분히 설명
- `layout_variance=3`: 안정적 보고서형 레이아웃 우선
- `content_preservation_priority=max`: 형식보다 내용 보존 우선
- `execution_clarity_priority=max`: 누가/언제/무엇을 해야 하는지 우선

## Step 1.5. 콘텐츠 분석 (내부 — 출력 안 함)

Step 2 구성 전 아래 3단계를 순서대로 수행한다. **컴포넌트는 마지막에 결정한다.**

### ① 사실 추출 — delta 고정값

| 항목 | 처리 |
|-----|------|
| 수치·달성률·퍼센트 | 원본 그대로 고정 |
| 고유명사·시스템명·제품명 | 원본 그대로 고정 |
| 날짜·버전·담당자명 | 원본 그대로 고정 |
| 동사·조사·연결어 | 슬라이드 형식 맞게 최소 조정 허용 |
| 원본에 없는 사실 | 절대 추가 금지 |

### ② 덱 모드 확인 (컴포넌트보다 먼저)

먼저 아래를 확인한다.

| mode | 의미 | 기본 여부 |
|-----|------|---------|
| **operational_detail_report** | 누가/언제/무엇을 해야 하는지 자기완결로 자세히 설명 | **기본값** |
| **presentation** | 희소 카피·요약형 발표용 | 사용자가 명시 요청한 경우만 |

원칙:
- 사용자가 별도 요청하지 않으면 `operational_detail_report`로 본다.
- 따라서 슬라이드는 기본적으로 **자세히 설명하는 페이지**여야 한다.

### ③ 슬라이드 역할 결정 (내용이 먼저, 컴포넌트는 아직 결정하지 않음)

각 슬라이드가 청중에게 **무엇을 하게 만드는가**를 먼저 정한다.

| 역할 | 정의 | 판단 기준 |
|-----|------|---------|
| **설명** | 사실·현황·배경을 전달 | "이게 뭔지" 알려주는 슬라이드 |
| **비교** | 두 상태·대상을 나란히 보여줌 | Before/After, A vs B, 현행 vs 개선 |
| **결정 유도** | 청중이 판단하거나 승인해야 함 | 숫자·조건·결론이 핵심인 슬라이드 |
| **순서** | 단계나 프로세스를 따라가게 함 | 1→2→3 흐름이 있는 슬라이드 |
| **체크/확인** | 완료 여부나 상태를 확인하게 함 | 완료·진행중·대기 항목이 있는 슬라이드 |

역할을 먼저 확정하지 않으면 ③으로 넘어가지 않는다.

### ④ 역할 → 컴포넌트 매핑 (v6 원칙)

| 역할 | 우선 컴포넌트 | 비고 |
|-----|---------|------|
| 표지 | `cover()` | 항상 첫 슬라이드 |
| 설명 / 체크 / 결정 유도 | `slide()` | 기본값. 가장 자유도가 높음 |
| 순서 | `slide()` 또는 `flow()` | 내용이 길면 `slide()` 우선 |
| 비교 | `two_col()` | 진짜 비교일 때만 사용 |

**원칙**:
- 컴포넌트는 역할의 결과다. 컴포넌트가 역할을 강요하면 안 된다.
- 상세 설명이 필요하면 구조형 컴포넌트보다 `slide()` 같은 서술형을 우선한다.
- 발표용 희소 카피보다 운영용 자기완결 설명을 우선한다.

### ⑤ Voice Fingerprint 확인

생성 전 Spigen 문체 기준을 내부적으로 체크. 출력하지 않음.

- 헤더: **명사형 강제**, 10자 이내 권장. 동사 어미(`~한다` `~된다` `~만든다`) 금지.
  - O: "패키지 리뉴얼 방향" / "출하량 30% 감소" / "바커 자동화 흐름"
  - X: "패키지 리뉴얼 방향에 대한 검토" / "현황 분석 및 방향성 제시" / "바커를 자동화한다"
- bullet / 본문: 운영용은 5~6줄까지 허용
- 한 페이지에서 실행 조건, 상태, 주체, 다음 액션이 같이 보이면 오히려 좋다
- 실제 화면에서 보이는 컬럼명 / 상태명 / 버튼명 우선
- 내부 ID / 함수명 / boardId는 부록이 아니면 본문에 넣지 않음
- 수치가 있으면 수치 먼저, 설명은 다음
- AI 클리셰 없음: 혁신적인·원활한·극대화·시너지·솔루션·최적화 → 구체 수치·사실로 대체

## Step 2. 슬라이드 구성 제안

### 출력 형식

```
=== 슬라이드 구성 ===

[1] 표지 — (제목)
[2] (슬라이드 제목) — (핵심 내용 1줄)
[3] ...

승인하시면 생성을 시작합니다.
```

### 슬라이드 유형 선택

Step 1.5 ②에서 결정한 **역할**을 그대로 가져온다. 역할 없이 컴포넌트를 고르지 않는다.

| 역할 | 함수 |
|------|------|
| 표지 | `cover()` |
| 설명 / 순서 / 체크 / 결정 유도 | `slide()` |
| 비교 (Before/After, A vs B) | `two_col()` |

### 슬라이드 수 기준

| 목적 | 권장 장수 |
|------|----------|
| 빠른 보고 | 4~6장 |
| 교육·안내 | 6~10장 |
| 제안서 | 5~8장 |

### Step 3 진입 조건

- [ ] 사용자가 구성을 승인했는가
- [ ] 모든 슬라이드 내용이 사용자 제공 데이터에 근거하는가

하나라도 No → 구성 수정 후 재승인.

---

## 수정 요청 처리

**구성 단계 수정**: 해당 슬라이드만 수정 → 전체 구성 재출력 → 승인  
**생성 후 수정**: 해당 슬라이드만 재생성  
**생성 후 구조 변경**: Step 2부터 재시작
